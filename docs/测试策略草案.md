# MetaCode 测试策略草案

生成日期：2026-06-12

## 1. 文档目的

本文档用于整理 MetaCode 当前阶段的测试目标、测试分层、测试副作用和后续测试建设顺序。

当前项目已经推进到 Stage 14，功能上已经具备：

```text
能力单元 -> Workflow -> 失败诊断 -> 修复/规划 -> 显式 repair_id -> 修复事件链 -> Dashboard 触发与详情查看
```

下一阶段不优先继续扩展功能细节，而是先把测试体系整理清楚，让后续功能可以边做边验证。

## 2. 测试总目标

MetaCode 的测试不是只证明“代码没有报错”，而是证明四件事：

1. 能力单元可以稳定复用。
2. Workflow 可以被正确执行、失败和修复。
3. 失败诊断、修复生成、事件链绑定是结构化且可追踪的。
4. Dashboard 和 API 展示的是可信的工程状态。

测试应尽量回答：

```text
这个系统是否仍然保持最小工程闭环？
这次改动是否破坏了历史阶段能力？
测试产生的数据哪些是资产，哪些只是副作用？
```

## 3. 测试分层

### 3.1 单元测试

目标：验证核心函数的局部行为。

覆盖范围：

```text
core/context.py
core/recommender.py
core/workflow_fixer.py
core/path_planner.py
core/capability_graph.py
core/monitoring_store.py
```

重点检查：

```text
字段路径判断是否正确
missing_fields 是否能被识别
推荐 metacode 是否按 ready / score 输出
fixed workflow 是否插入正确步骤
planned workflow 是否能生成多步路径
监控聚合是否稳定排序
repair event 是否能正确绑定 failure 与 verification
```

### 3.2 Workflow 端到端测试

目标：验证真实 workflow 能跑通，失败样例能按预期失败。

分两类：

```text
稳定 workflow：必须 success
故意失败 workflow：必须 failed，并且包含 missing_fields / suggestions
```

当前代表样例：

```text
workflows/note_to_summary.yaml
workflows/note_full_report.yaml
workflows/broken_missing_clean_text.yaml
workflows/broken_missing_summary_chain.yaml
```

这类测试会写入 `logs/`，因此需要明确它们的副作用。

### 3.3 修复闭环测试

目标：验证失败能进入修复路径，并生成可验证的事件链。

覆盖范围：

```text
fix_workflow
plan_workflow_fix
POST /api/repair/fix-workflow
POST /api/repair/plan-workflow
repair_events.json
```

最低要求：

```text
fixed 修复能生成 generated workflow
planned 修复能生成多步 generated workflow
API 触发修复时返回显式 repair_id
repair_id 能绑定 failure run 和 verification run
repair event 状态能闭环到 closed_success 或 closed_failed
```

### 3.4 API 测试

目标：验证 Dashboard 所依赖的数据接口结构稳定。

核心接口：

```text
GET /api/status
GET /api/monitoring
GET /api/runs
GET /api/failures
GET /api/workflows
GET /api/diagnostics/*
GET /api/repairs/*
GET /api/repair-events/*
POST /api/repair/fix-workflow
POST /api/repair/plan-workflow
```

API 测试重点不是断言所有数值完全固定，而是断言：

```text
字段存在
状态合法
数量关系一致
排序稳定
阶段版本向前兼容
```

例如旧阶段测试不应强制要求 `version == stage13`，而应允许 `stage13` 或更新阶段。

### 3.5 Dashboard 测试

当前先保持轻量，不引入复杂前端测试框架。

最低测试：

```powershell
node --check dashboard\app.js
```

静态结构检查：

```text
关键 section 是否存在
关键容器 id 是否存在
关键函数名是否存在
关键 CSS class 是否存在
```

人工浏览器检查：

```text
Dashboard 能加载
API 模式和静态回退模式都可读
修复按钮只在 API 模式可触发
点击最近事件能显示详情
桌面和移动端没有明显横向溢出
```

## 4. 测试副作用规则

当前最重要的问题：测试会刷新运行日志和监控导出。

会被测试改变的文件包括：

```text
logs/run_log.jsonl
logs/failure_log.jsonl
monitoring/exports/dashboard_summary.json
monitoring/exports/runs.json
monitoring/exports/failures.json
monitoring/exports/repair_metrics.json
monitoring/exports/repair_events.json
monitoring/exports/stages.json
monitoring/stages/stage_*.json
```

建议先采用两类测试约定：

### 4.1 项目状态验证

这类测试允许写入真实 `logs/` 和 `monitoring/exports/`。

用途：

```text
阶段完成前的最终验证
更新 Dashboard 真实展示数据
生成阶段报告所需指标
```

运行时机：

```text
阶段收尾
提交前
用户明确要求刷新项目状态时
```

### 4.2 隔离行为验证

这类测试应该尽量使用临时目录或临时日志文件。

用途：

```text
验证函数行为
验证边界条件
避免每次测试污染项目历史日志
```

后续需要逐步改造的方向：

```text
让 execute_workflow 支持可配置 log_dir
让 export_monitoring_data 支持输入临时 root
测试 fixtures 复制最小 workflow / metacode 子集
API 测试使用临时项目副本
```

短期不必一次性完成这些改造，但新测试应优先考虑隔离副作用。

## 5. 推荐测试命令

阶段开发中，优先跑局部测试：

```powershell
python -m unittest tests.test_stage14_unittest -v
node --check dashboard\app.js
```

阶段收尾时，跑完整验证：

```powershell
python export_monitoring_data.py
python -m unittest discover -s tests -p "*unittest.py" -v
node --check dashboard\app.js
```

如果改动 Dashboard 交互，还需要启动本地服务人工检查：

```powershell
python serve_observatory.py --port 8770
```

访问：

```text
http://127.0.0.1:8770/dashboard/
http://127.0.0.1:8770/api/status
```

## 7. 现有测试分层清单

当前先把测试分成三类，后续新增测试应优先选择“隔离行为验证”。

### 7.1 隔离行为验证

这类测试不写真实日志，不刷新监控导出，适合开发中高频运行。

```text
tests/test_stage15_unittest.py
```

当前覆盖：

```text
repair event 显式 repair_id 绑定
Stage 13 旧推断逻辑兼容
```

推荐命令：

```powershell
python -m unittest tests.test_stage15_unittest -v
```

### 7.2 轻副作用或结构检查

这类测试主要检查结构、文档、纯函数或前端文件存在性，通常不刷新监控导出，但可能读取项目文件。

```text
tests/test_stage3_unittest.py
tests/test_stage6_unittest.py
tests/test_stage8_unittest.py
```

说明：

```text
如果测试中调用 execute_workflow，则仍会写 logs/。
如果只读 dashboard/docs/core 结构，则可视为轻副作用。
```

### 7.3 项目状态验证

这类测试会写真实 `logs/`、生成 workflow，或刷新 `monitoring/exports/`。它们适合阶段收尾和提交前运行，不适合每次小改都跑。

```text
tests/test_prototype_unittest.py
tests/test_stage2_unittest.py
tests/test_stage4_unittest.py
tests/test_stage5_unittest.py
tests/test_stage7_unittest.py
tests/test_stage9_unittest.py
tests/test_stage10_unittest.py
tests/test_stage11_unittest.py
tests/test_stage12_unittest.py
tests/test_stage13_unittest.py
tests/test_stage14_unittest.py
```

主要副作用来源：

```text
execute_workflow -> logs/run_log.jsonl 或 logs/failure_log.jsonl
fix_workflow -> workflows/generated/*.fixed.yaml + logs/
plan_workflow_fix -> workflows/generated/*.planned.yaml + logs/
export_monitoring_data -> monitoring/exports/*.json + monitoring/stages/*.json + SQLite
POST /api/repair/* -> logs/ + generated workflow + monitoring exports
```

### 7.4 推荐运行策略

开发中：

```powershell
python -m unittest tests.test_stage15_unittest -v
node --check dashboard\app.js
```

阶段收尾：

```powershell
python -m unittest discover -s tests -p "*unittest.py" -v
node --check dashboard\app.js
```

## 8. 下一步整理顺序

建议下一个工作目标不是新增功能，而是按以下顺序整理测试：

1. 先完成一次全量测试，记录当前失败点。
2. 把“旧阶段版本写死”的测试改成向前兼容。
3. 标记哪些测试会写真实日志和导出文件。
4. 为核心修复逻辑补少量隔离测试。
5. 再决定是否引入临时项目副本测试模式。

暂时不建议立刻做：

```text
复杂权限测试
多用户并发测试
远程部署测试
完整浏览器自动化测试体系
大规模重构测试目录
```

## 9. 阶段边界建议

建议把下一阶段暂定为：

```text
Stage 15：测试体系整理与验证基线
```

Stage 15 的完成标准可以先定得很小：

```text
全量测试可稳定通过
测试副作用被文档化
旧阶段测试支持阶段升级
至少一部分核心测试具备隔离运行能力
Dashboard 语法检查和人工检查流程明确
```

这样后续再继续做修复审核、候选路径比较、生成 workflow 提升等功能时，项目不会因为测试边界不清而变得难以判断。
